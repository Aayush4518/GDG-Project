**Slide 1 — Title & Project Detail**
- **Title:** TravelGuardian — Smart Tourist Safety & Incident Response Dashboard
- **Subtitle / Short detail:** Real-time monitoring dashboard for tourist safety with AI anomaly detection, geofencing alerts, live map visualization and blockchain-backed incident verification. Built as a React + Vite single-page app for authorities (police / tourism) to monitor, verify and respond to incidents.

**Slide 2 — Introduction**
- **Project overview:** TravelGuardian is a web dashboard prototype that aggregates live location and alert data for tourists, provides AI-style anomaly detection visualizations, and offers operational tools (assign units, generate E-FIRs, export CSV) to speed emergency response.
- **Audience:** Police, Tourism Authorities, Incident Response Teams, and Developers for demonstration.
- **Demo mode:** The dashboard loads large dummy datasets (500+ tourists and alerts) for demonstration, and supports WebSocket-driven live updates.

**Slide 3 — Problem Statement**
- **Context:** Tourist safety in high-traffic or remote areas is challenging due to delayed detection and coordination gaps among agencies.
- **Problems addressed:**
	- Lack of centralized live situational awareness for authorities
	- Slow detection of unusual behavior or panic (manual monitoring)
	- Difficulty coordinating fast dispatch and recording tamper-proof incident logs

**Slide 4 — Objectives**
- **Primary objective:** Provide a single-pane-of-glass dashboard that accelerates detection, verification and response for tourist-related incidents.
- **Secondary objectives:**
	- Demonstrate real-time updates via WebSocket
	- Provide exportable records (CSV, E-FIR PDF)
	- Support digital ID verification for tourists
	- Showcase map visualizations and large dummy-data handling for demos

**Slide 5 — Proposed Solution (High level)**
- **Solution summary:** A React SPA that:
	- Loads large demo datasets (massive_tourists.json / massive_alerts.json) for realistic demo scenarios
	- Uses WebSocket to receive live alerts and location updates, updating UI instantly
	- Presents interactive maps (Leaflet / custom markers, 3D map component stub) and trend visualizations
	- Provides operational tools: assign units, send help, create E-FIR, ledger verification endpoint

**Slide 6 — Tech Stack (concise & simple)**
- **React + Vite:** UI framework and fast dev/build tooling. React builds the interactive dashboard; Vite runs the dev server and builds production bundles.
- **Leaflet + react-leaflet:** Map rendering and markers for live location, geofencing and risk zones.
- **axios + websocketService:** `axios` for REST API calls; a small `websocketService` for real-time alert/location updates.
- **PostgreSQL:** Relational database (recommended) to store tourists, alerts, incident logs, and ledger records in a production setup.
- **Static demo files:** JSON files in `public/` (massive_tourists.json, massive_alerts.json, analytics.json) used for offline/demo mode.
- **file-saver + localStorage:** Client-side tools for exporting CSV/PDF and persisting simple auth/theme settings.
- **Dev tools:** ESLint for linting and Vite plugins for React support.

*Note:* The current repo is a frontend prototype that expects backend services (APIs + DB). PostgreSQL is suggested for production storage and ledger records.

**Slide 7 — Key Features (mapped to files/components)**
- **Live Map & Markers:** `src/components/Advanced3DMap.jsx` + `leaflet` in `src/App.jsx` — shows tourists, alerts and risk zones.
- **Live Activity Feed:** `src/components/LiveActivityFeed.jsx` — streaming alerts and notifications.
- **Alerts Table & Filters:** Implemented inside `src/App.jsx` (AlertsTable, Filters) — filtering by district and severity, pagination.
- **E-FIR & Export:** `EFIRForm`, `exportCSV` — generate PDFs and CSV downloads using `apiService` and `file-saver` pattern.
- **Digital ID Verification:** `VerifyID` section uses `public/ids.json` to look up tourist IDs.
- **Performance & Monitoring:** `src/components/PerformanceMonitor.jsx` — monitors websocket/API health and shows connection status.

**Slide 10 — Data Flow (textual diagram)**
- **Data sources:**
	- Static demo files in `public/` (`massive_tourists.json`, `massive_alerts.json`, `analytics.json`, `ids.json`)
	- External backend via `apiService` (risk zones, ledger verification, tourist details)
	- Real-time stream via `websocketService` (events: LOCATION_UPDATE, PANIC_ALERT, ALERT_RESOLVED, etc.)
- **Flow steps:**
	1. On load, app fetches demo JSON files from `public/` and populates `tourists`, `alerts`, `analytics` state.
	2. `websocketService.connect()` opens a WebSocket -> incoming events processed by `handleWebSocketMessage` handler.
	3. Real-time events update `tourists`, `alerts`, and `activeAlerts` state; UI components re-render.
	4. User actions (Assign Unit, Send Help, Create E-FIR) call `apiService` endpoints (POST /generateEFIR, POST /assign, etc.).
	5. Export and verification tools use `file-saver` (client) or `apiService` responses to provide CSV/PDF downloads and ledger confirmations.
